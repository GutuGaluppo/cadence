# Release Playbook

This document closes the operational gap between a working repository and a public downloadable release.

## 1. Configure the repository once

### GitHub Pages

The repository now includes `.github/workflows/pages.yml`, which deploys the `site/` directory to GitHub Pages.

Repository setting required:

1. Open `Settings > Pages`.
2. Under `Build and deployment`, set `Source` to `GitHub Actions`.

This workflow follows GitHub’s official Pages guidance using `actions/configure-pages@v5`, `actions/upload-pages-artifact@v4`, and `actions/deploy-pages@v4`.

Reference:

- https://docs.github.com/en/pages/getting-started-with-github-pages/using-custom-workflows-with-github-pages

### Release notes

The repository now includes `.github/release.yml` so GitHub can group autogenerated release notes by label when `generateReleaseNotes: true` runs in the release workflow.

## 2. Configure release secrets

### Required for all signed updater artifacts

Tauri’s updater signing requires a private key in CI and `createUpdaterArtifacts: true` is already enabled in `src-tauri/tauri.conf.json`.

Generate the key locally:

```bash
pnpm tauri signer generate -w ~/.tauri/cadence.key
```

Add these GitHub repository secrets:

- `TAURI_SIGNING_PRIVATE_KEY`
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`

Reference:

- https://v2.tauri.app/ko/plugin/updater/

### Required for signed and notarized macOS releases

The release workflow now supports importing a base64-encoded Apple certificate into a temporary keychain on macOS runners. To activate that path, configure:

- `APPLE_CERTIFICATE`
- `APPLE_CERTIFICATE_PASSWORD`
- `KEYCHAIN_PASSWORD`
- `APPLE_ID`
- `APPLE_PASSWORD`
- `APPLE_TEAM_ID`

Optional:

- `APPLE_SIGNING_IDENTITY`

If `APPLE_SIGNING_IDENTITY` is not provided, the workflow will try to resolve it from the imported certificate automatically.

References:

- https://v2.tauri.app/ko/distribute/sign/macos/
- https://v2.tauri.app/es/reference/environment-variables/

### Optional Linux signing

If RPM signing becomes part of the release contract, also configure:

- `TAURI_SIGNING_RPM_KEY`
- `TAURI_SIGNING_RPM_KEY_PASSPHRASE`

### Windows signing

Windows release assets are already built by the matrix, but certificate-backed signing still depends on certificate-specific metadata or a custom sign command. That should be finalized only after the certificate vendor and signing route are chosen.

Reference:

- https://v2.tauri.app/ko/distribute/sign/windows/

## 3. Prepare a release

Before tagging:

1. Run `pnpm test`.
2. Run `pnpm build`.
3. Run `cargo fmt --manifest-path src-tauri/Cargo.toml --all --check`.
4. Run `cargo clippy --manifest-path src-tauri/Cargo.toml --all-targets -- -D warnings`.
5. Review `CHANGELOG.md` and public site copy.

## 4. Publish a release

Tag and push a semantic version:

```bash
git tag v0.1.0
git push origin v0.1.0
```

That triggers `.github/workflows/release.yml`, which:

- builds Tauri bundles for macOS, Linux, and Windows
- publishes assets to GitHub Releases
- generates release notes automatically
- uploads per-platform checksum files

## 5. Verify after publishing

Check the following immediately after the workflow succeeds:

1. GitHub Release exists for the tag.
2. Checksums were uploaded for each platform that produced assets.
3. The landing page download button opens the latest public release.
4. The GitHub Pages deployment for `site/` is green.
5. macOS assets are signed and notarized if Apple secrets were configured.
